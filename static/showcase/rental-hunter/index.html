机器学习 异常检测 GradientBoosting

第一章：那些不对劲的房子

假设你在北京找房子。打开贝壳，刷到一条房源：朝阳区，两居室，68㎡，距地铁 500 米——月租 3,200 元。

便宜吗？周围类似条件的房子挂 5,500 起步。你心里嘀咕：是房东急租，还是信息有猫腻？

这个直觉，恰好是异常检测的核心思想：如果一个事物的"标价"和它"应该值多少"之间差距太大，就值得怀疑。

任务设定：一份真实的北京租房数据集（40,000 条训练集），其中 测试集的 1,000 条房源里混入了 100 个被人为篡改了价格的"假房源"——有的虚减 10%~40%，有的虚增 10%~40%。找出它们。

问题是——你不知道哪些被篡改了，也不知道篡改方向和幅度。你只有一堆房源的特征：面积、楼层、户型、区位、地铁距离，以及各种标签。

第二章：给每个房子算一个"合理价"

人的直觉能覆盖多少？6 条，也许 60 条。但 1,000 条房源——你需要一个系统方法。

核心假设很简单：

如果一个房源的真实租金可以被它的其他特征（面积、区位、楼层、户型……）较准确地预测，那么被篡改过价格的房源，其实际价格和模型预测的"合理价格"之间，会出现明显偏差。

方法论只有一句话：训练一个租金预测器 → 用它算每个房源"应该值多少" → 实际价格偏离预测越大的，越可疑。

但魔鬼在细节里。

特征工程：哪些信息决定了租金？

原始数据有 30+ 个字段。经过筛选和组合，最终喂给模型的特征分三类：

面积、楼层、经纬度…
4

区、整/合租、平台…
5
衍生特征
每卧面积、楼层比…

衍生特征是从原始字段计算出来的。比如 area_per_bedroom = rent_area / bedroom_num，捕捉的是"这个房子每间卧室有多大"——同样 80㎡，两居和四居的租金天差地别。

为什么选 GradientBoosting？

租金和特征之间的关系不是线性的。"朝阳区 + 大面积"的组合效应、"高楼层在老小区可能是缺点但在新楼盘是加分项"——这些非线性关系和交互效应，线性回归搞不定。

GradientBoosting 本质上是一棵棵决策树依次纠错叠加：第一棵树拟合数据，第二棵树拟合第一棵的残差，第三棵再拟合剩余残差……几百棵弱学习器叠起来，就能捕捉非常复杂的模式。

第三章：残差——偏离就是证据

模型训练完成后，对测试集的每条房源预测一个"合理价格"。然后算一个简单的东西——相对残差：

residual = (实际价格 − 预测价格) / 预测价格
正值意味着实际贵于预期（可能虚增），负值意味着便宜于预期（可能虚减）

为什么用"相对"而不是"绝对"残差？因为一套月租 20,000 的房子偏差 2,000 元和一套月租 3,000 的房子偏差 2,000 元，严重程度完全不同。

把所有房源按残差排序——最负的那些，最可能是被虚减的；最正的那些，最可能是被虚增的。

PYTHON 
# 预测 → 还原 → 算相对残差
pred_log = model.predict(X_test)
pred_price = np.expm1(pred_log)

residual = (actual_price - pred_price) / pred_price

# 虚减任务：残差最负的 top-N 最可疑
suspects_down = residual.nsmallest(120).index

# 虚增任务：残差最正的 top-N 最可疑
suspects_up = residual.nlargest(120).index

提交 100 还是 120？精度与召回的博弈

真实异常有 100 个。如果我提交恰好 100 个 house_id，命中率上限就被锁死了——排在第 101~120 名的可能还有真异常，但永远不会被捞到。

多提交 20 个会怎样？精度会略降（分母变大），但召回可能显著提升（多捞回了一些排在 100 名之外的真异常）。

评价指标是 TPR − FPR。数学上可以证明：只要第 101~120 名的命中率还高于基础异常率（100/1000 = 10%），扩大提交范围就是赚的。而模型排序后这个区间的命中率远超 10%——所以 120 优于 100。

这就是异常检测中典型的 precision-recall tradeoff。不同评价指标对"扩大范围"的容忍度不同：TPR−FPR 鼓励你"宁可多报"，F1 要求你"报得准"。选择多少取决于你被什么指标考核。

第五章：复盘

这个项目在技术上并不复杂——一个 GBR 模型，一个残差排序，核心代码不超过 50 行。但它让我理解了几件事：

选择评价指标比优化模型更重要。 同一个模型，在 F1 和 TPR−FPR 下的最优提交策略完全不同。在真实业务中，"我在优化什么"永远是第一个要回答的问题。

特征工程是性价比最高的事。 花 30 分钟构造 area_per_bedroom 和 floor_ratio，效果超过花 2 小时调超参。模型能学多好，上限由数据决定。

log 变换不是技巧，是对数据分布的尊重。 不理解数据的分布形态就建模，模型会悄悄地偏向某类样本而你毫无知觉。

如果有更多时间，可以尝试 LightGBM + GBR 的集成、加入小区名均价统计特征、或者用 Isolation Forest 做对比实验。但在课堂竞赛的投入产出比考量下，这些留给下次了。
