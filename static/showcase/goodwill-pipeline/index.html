藏在附注深处的炸弹

3,617 份年报，63,000 次关键词命中，922 笔商誉减值。 从非结构化的 PDF 附注里，把 A 股的"并购后遗症"挖出来。

RPA + NLP PDF 解析 正则分类器

第一章：并购潮退去之后

2015-2017 年，A 股经历了一轮空前的并购浪潮。上市公司以高溢价收购资产，账面上形成了大量"商誉"——本质上是"为这笔收购多付的钱"。

当被收购公司业绩达不到预期，这些商誉就需要"减值"——直接从利润里扣除，有时一笔减值就能吞掉全年利润。2018-2019 年 A 股集中爆发了商誉减值潮，一些公司一夜亏损数十亿。

几年过去了，减值压力消退了吗？哪些公司还在持续"消化"当年的并购遗产？它们的减值测试假设是否合理？

目标：从 3,617 份 2023 年度年报中，自动化提取商誉变动数据、减值金额和 DCF 测试参数——这些信息全部藏在年报附注的非结构化文本里，人工翻阅几乎不可能。

第二章：为什么这件事很难

商誉信息不在年报的标准化字段里。它散落在 PDF 附注的第 80 页、第 127 页，混杂在其他减值测试、会计政策描述和资产负债表行项之间。

每家公司的年报格式不同——表格结构、附注编号、金额单位（元/万元/百万元）都可能不一样。有的公司年报是扫描版（纯图片），连文字都提取不出来。

更棘手的是噪声。搜索"商誉"这个关键词，几乎每份年报都会命中——但绝大部分命中是资产负债表里的一行科目名，或者会计政策模板里的标准化描述。真正有分析价值的段落（变动表、减值测试参数、减值金额）只占一小部分。

这意味着：你不仅需要"找到提到商誉的地方"，还需要"判断这个提及是否有信息量"。前者是搜索问题，后者是分类问题。

第三章：五阶段 Pipeline

整个系统分五个阶段，每个阶段的输出都是独立文件，中断后可从任意阶段恢复。

为什么不直接用 LLM？

3,617 份年报、每份几百页——全部送进 LLM 的成本和延迟都不可接受。而且这个任务的核心不是"理解语义"（每份年报的表述其实高度模板化），而是"在海量文档中精确定位并提取结构化数据"。传统正则在这里比 LLM 更快、更可控、更可调试。

当然，对于更复杂的提取任务（比如从自由文本中理解减值测试的推理逻辑），LLM 的优势就体现出来了。这个 pipeline 的设计是"传统方法能解决的交给正则，LLM 留给真正需要语义理解的环节"。

第四章：信号与噪声

搜索"商誉"和"减值测试"两个关键词，在 3,617 份年报中产生了 63,437 次原始命中。但并非每次命中都有信息量——事实上，大部分命中是噪声。

★ 标记的三类是高价值命中。"资产负债表"是年报里列出的那行科目名（`商誉 5,408,770,448.41`），几乎每家公司都有，但不含任何分析信息。"会计政策"是附注里关于商誉确认方法的模板文字，各公司大同小异。

分类器是怎么工作的？

五类命中各有不同的文本特征，用正则匹配来判断。比如资产负债表行项的典型特征是"商誉"后紧跟制表符和数字（商誉\t19\t5,408...），或者上下文包含"非流动资产合计"。

分类器的优先级是：减值金额 > 减值测试 > 变动表 > 会计政策 > 资产负债表。这样即使一段文本同时匹配多个类别，也会被归入信息量最高的那个。

第五章：两个关键技术决策

表格结构保持：坐标聚类

PDF 里的表格不是"表格"——它们只是一堆浮在页面上的文本块，每个块有自己的 (x, y) 坐标。直接提取文本会丢失所有表格结构，商誉变动表会变成一串无法解析的数字流。

PYTHON 
# 核心思想: 同一行的文本块，y 坐标应该接近
# 按 y 聚类 (3px 容差) → 行内按 x 排序 → 制表符连接

spans.sort(key=lambda s: (s["y"], s["x"]))

for span in spans[1:]:
    if abs(span["y"] - current_cluster[-1]["y"]) <= 3.0:
        current_cluster.append(span)  # 同一行
    else:
        current_cluster.sort(key=lambda s: s["x"])
        row = "\t".join(s["text"] for s in current_cluster)
        rows.append(row)  # 换行

这个方法的好处是对格式差异有一定容忍度——只要同一行的文本 y 坐标差距在 3px 以内，就能正确聚合。

上下文窗口：前 3 + 后 25 行

关键词命中只告诉你"第 247 行有'商誉'"。但减值金额可能在第 250 行，被投资方名称可能在第 248 行。不对称的上下文窗口（前 3 行 + 后 25 行）是经验调参的结果：商誉变动表通常在关键词之后展开，所以向下多取。

折现率：估值假设是否合理？

商誉减值测试的核心是 DCF 估值——预测未来现金流，用折现率折回现值。折现率越低 → 现值越高 → 越不容易触发减值。因此管理层在选择折现率时有动机"偏乐观"。

潜在风险：商誉>1亿但未减值

289 家公司持有超过 1 亿元的商誉，但 2023 年度未计提任何减值。这不一定意味着有问题——如果被收购公司业绩良好，不减值是正确的。但如果未来业绩恶化，这些商誉可能集中释放减值压力。

第七章：局限与反思

金额提取精度有限。 正则表达式在高度模板化的文本上表现不错，但 A 股年报的格式多样性超出预期。3,617 家公司中只有 1,518 家成功提取到商誉金额，覆盖率约 42%。减值发生率（922/1,518 ≈ 60.7%）可能因分母偏小而虚高——实际有商誉的公司可能远多于 1,518 家。

单位未统一。 不同公司使用元、万元、百万元为单位，当前版本未做自动换算。Top 10 减值表格中的数字单位以各公司原始披露为准——这意味着跨公司比较需要额外处理。

如果重来，我会：

在阶段 3 增加单位检测（从附注标题提取"单位：万元"），自动换算到统一单位。在阶段 4 对高价值命中采用 LLM 辅助提取（用 Claude API 处理正则难以覆盖的非标准格式），作为规则引擎的补充而非替代。以及加入行业维度分析——商誉减值在不同行业的分布应该有很强的聚集效应（传媒、游戏、医疗是重灾区）。

但这些留给下一个版本了。这个项目让我理解的核心是：数据工程的难点不在算法，而在对数据本身的理解。PDF 里的表格不是表格，关键词命中不等于有效信息，分类器的设计取决于你对数据噪声模式的认知。这些认知只能从亲手翻阅几十份年报、逐条 debug 正则中获得——没有捷径。
